# Contributing

Contributions should make ServerDi Topic OS clearer, safer, and easier to use in public.

## Ground Rules

- Keep each change focused on one bounded outcome.
- Do not include private deployment details, raw logs, sessions, memory, backups, env values, or real identifiers.
- Use placeholders for Telegram IDs, workspaces, model names, and approval IDs.
- Add validation notes when changing process guidance or prompt templates.
- Keep language practical and operational.

## Good Contribution Types

- clearer micro-pass templates;
- safer prompt wording;
- additional validation checklists;
- public-safe examples;
- architecture notes that avoid private deployment specifics;
- corrections to ambiguous boundary rules.

## Pull Request Checklist

- [ ] The change has one clear outcome.
- [ ] No secrets, tokens, auth files, cookies, passwords, or private keys are included.
- [ ] No real Telegram IDs, private paths, server IPs, logs, sessions, memory dumps, backups, or runtime state are included.
- [ ] Examples use placeholders only.
- [ ] Documentation remains useful without depending on private infrastructure.
- [ ] The final report explains what changed and what was not touched.
