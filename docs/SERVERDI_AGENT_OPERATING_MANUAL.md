# ServerDi Agent Operating Manual

This manual defines a small operating model for safe AI agent work. It is written for public use and intentionally avoids private deployment details.

## Operating Rule

One task equals one bounded outcome. A pass must finish with either a validated result, a clear blocker, or a recommendation for the next bounded pass.

Avoid "fix everything" requests. They create broad authority, unclear review scope, and weak validation.

## Pass Types

### Read-Only Scout

Purpose: inspect context and report findings.

Allowed:

- read public repository files;
- inspect documentation structure;
- identify missing checks or unclear boundaries;
- recommend one next bounded task.

Not allowed:

- editing files;
- changing configuration;
- calling paid tools;
- touching runtime state;
- restarting services.

### Patch Pass

Purpose: make one scoped change and validate it.

Required:

- state intended files before editing;
- keep changes narrow;
- validate with concrete checks;
- report changed files and untouched areas.

### Audit Pass

Purpose: review a patch, prompt, document, or process for safety and correctness.

Output:

- findings ordered by severity;
- file or section references when available;
- residual risk;
- recommended next step.

## Forbidden Actions

Agents must not perform these actions unless a separate owner-approved operational procedure explicitly permits them:

- access secrets, tokens, env values, auth files, cookies, passwords, or private keys;
- publish real Telegram IDs or topic IDs;
- inspect private sessions, memory dumps, backups, raw logs, generated media, or runtime state;
- copy private workspace files into public docs;
- change Docker, OpenClaw, Hermes, Telegram, or production configuration as a side effect of documentation work;
- restart production services without explicit approval;
- bypass approval gates for paid or risky tools.

## Validation-First Workflow

1. Define the bounded outcome.
2. Confirm allowed scope.
3. Run the pass.
4. Validate using checks appropriate to the task.
5. Inspect any dangerous terms in public files.
6. Commit only after validation passes.
7. Report the result with evidence.

Validation evidence can include:

- file tree;
- git diff summary;
- status output;
- test output;
- rendered documentation;
- successful delivery or event confirmation.

A local file generated by an agent is not proof of delivery by itself.

## Final Report Format

Use this structure:

```text
Outcome:
Changed:
Not touched:
Validation:
Sanitization:
Commit:
Push:
Blockers:
Next recommended pass:
```

## Rollback Decision Rules

Rollback is appropriate when:

- validation fails and the fix is unclear;
- a change crosses the approved scope;
- public files contain private or unsafe data;
- an operational side effect happened unexpectedly.

Rollback is not required when:

- a validation warning is understood and documented;
- a dangerous term appears only in a safety warning;
- the next fix is small and within the same bounded outcome.

## No Broad Cleanup Rule

Do not combine documentation creation, dependency updates, formatting churn, runtime cleanup, and deployment changes in one pass. Split them into separate tasks.
